Types¶

Execution has to check and compare types and type instances in a few places, such as executing \(\href{../syntax/instructions.html#syntax-instr-control}{\mathsf{call\_indirect}}\) or instantiating modules. During execution, types of all forms are represented as dynamic types, where all occurring type identifiers are interpreted as type addresses. Relevant type relations need to be redefined accordingly.

Note

Runtime type checks generally involve types from multiple modules or types not defined by a module at all, such that module-local type indices are not meaningful. Type addresses are global to a store and can hence be interpreted independent of module boundaries.

Type Identifiers¶

During execution, type identifiers are represented as type addresses, which are looked up as function types in the store by the following rule.

\(\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}\)¶

  • The type \(S.\href{../exec/runtime.html#syntax-store}{\mathsf{types}}[\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}]\) must be defined in the store.

  • Then the type address is valid as function type \(S.\href{../exec/runtime.html#syntax-store}{\mathsf{types}}[\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}]\).

\[\frac{ S.\href{../exec/runtime.html#syntax-store}{\mathsf{types}}[\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}] = \href{../syntax/types.html#syntax-functype}{\mathit{functype}} }{ S; C \href{../valid/types.html#valid-typeid}{\vdash} \href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}} : \href{../syntax/types.html#syntax-functype}{\mathit{functype}} }\]

Note

Unlike type indices recorded in a context, the number of type addresses in a store is not bounded by \(2^{32}\).

Instantiation¶

Any form of static type can be instantiated into a dynamic type inside a module instance by substituting each type index \(x\) occurring in it with the corresponding type address \(\href{../exec/runtime.html#syntax-moduleinst}{\mathit{moduleinst}}.\href{../exec/runtime.html#syntax-moduleinst}{\mathsf{types}}[x]\).

\[\href{../exec/types.html#dyn}{\mathrm{dyn}}_{\href{../exec/runtime.html#syntax-moduleinst}{\mathit{moduleinst}}}(t) = t[\href{../syntax/conventions.html#notation-subst}{\mathrel{\mathbf{:=}}} \href{../exec/runtime.html#syntax-moduleinst}{\mathit{moduleinst}}.\href{../exec/runtime.html#syntax-moduleinst}{\mathsf{types}}]\]

Dynamic Typing¶

To handle dynamic types, all static judgements \(C \vdash \mathit{prop}\) on types (such as validity and matching) are generalized to include the store, as in \(S; C \vdash \mathit{prop}\), by implicitly adding a store \(S\) to all rules – \(S\) is never modified by the pre-existing rules, but it is accessed in the extra rule for type addresses given above.

It is an invariant of the semantics that all types inspected by execution rules are dynamic, i.e., the context is always empty and never used. To avoid unnecessary clutter, empty contexts are omitted from the rules, writing just \(S \vdash \mathit{prop}\).

Note

Only matching rules are invoked during execution. Dynamic validity is only needed to prove type soundness and for specifying parts of the embedder interface.
