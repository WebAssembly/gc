Types¶

Execution has to check and compare types and type instances in a few places, such as executing \(\href{../syntax/instructions.html#syntax-instr-control}{\mathsf{call\_indirect}}\) or instantiating modules. During execution, types of all forms are represented as semantic types, where all occurring type identifiers are interpreted as type addresses. Relevant type relations need to be redefined accordingly.

Note

Runtime type checks generally involve types from multiple modules or types not defined by a module at all, such that module-local type indices are not meaningful. Type addresses are global to a store and can hence be interpreted independent of module boundaries.

Type Identifiers¶

During execution, type identifiers are represented as type addresses, which are looked up as function types in the store by the following rule.

\(\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}\)¶

  • The type \(S.\href{../exec/runtime.html#syntax-store}{\mathsf{types}}[\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}]\) must be defined in the store.

  • Then the type address is valid as function type \(S.\href{../exec/runtime.html#syntax-store}{\mathsf{types}}[\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}]\).

\[\frac{ S.\href{../exec/runtime.html#syntax-store}{\mathsf{types}}[\href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}}] = \href{../syntax/types.html#syntax-functype}{\mathit{functype}} }{ S \href{../valid/types.html#valid-typeid}{\vdash} \href{../exec/runtime.html#syntax-typeaddr}{\mathit{typeaddr}} : \href{../syntax/types.html#syntax-functype}{\mathit{functype}} }\]

Note

Unlike type indices recorded in a context, the number of type addresses in a store is not bounded by \(2^{32}\).

Instantiation¶

Any form of syntactic type can be instantiated into a semantic type inside a module instance by substituting each type index \(x\) occurring in it with the corresponding type address \(\href{../exec/runtime.html#syntax-moduleinst}{\mathit{moduleinst}}.\href{../exec/runtime.html#syntax-moduleinst}{\mathsf{types}}[x]\).

\[\href{../exec/types.html#sem}{\mathrm{sem}}_{\href{../exec/runtime.html#syntax-moduleinst}{\mathit{moduleinst}}}(t) = t[\href{../syntax/conventions.html#notation-subst}{\mathrel{\mathbf{:=}}} \href{../exec/runtime.html#syntax-moduleinst}{\mathit{moduleinst}}.\href{../exec/runtime.html#syntax-moduleinst}{\mathsf{types}}]\]

Matching and Validity¶

For each static matching relation on syntactic types, operating relative to a context, an analogous dynamic validity or matching relation is defined, operating relative to a store.

Formally, for each judgement

\[C \vdash T_1 \leq T_2\]

on syntactic types \(T_1\) and \(T_2\), an analogous judgement

\[S \vdash T'_1 \leq T'_2\]

on corresponding semantic types \(T'_1\) and \(T'_2\) is introduced. It is defined analogously, by replacing all occurrences of a context \(C\) in the associated rules with a store \(S\).

Note

Where the static rules invoke static lookup for type indices in the context, the dynamic rules thereby invoke dynamic lookup for type addresses in the store.

Likewise, for each static validity rule on syntactic types, operating relative to a context, an analogous dynamic validity rule is defined, operating relative to a store.

Formally, for each judgement

\[C \vdash T \mathrel{\mbox{ok}}\]

on syntactic type \(T\), an analogous judgement

\[S \vdash T' \mathrel{\mbox{ok}}\]

on corresponding semantic types \(T'\) is introduced. It is defined analogously, by replacing all occurrences of a context \(C\) in the associated rules with a store \(S\).

Note

Dynamic validity rules are not needed for the semantics, but to prove soundness and for specifying parts of the embedder interface.
